# Error Hunter

## 业务目标

Error Hunter 用于批量审核 AI 生成的面试题。它要还原真实审题业务流程：先用多个模型分别审题，再汇总每个模型的删除结论、错误类型和删除原因；没有被审出错误的题进入下一轮复审；被审出错误的题按“保留题干重出答案”和“题目答案都重出”分流，交给下游重新生成；重新生成后的题目或答案再回流进入三到 N 轮审核。

本 Skill 不负责真正重新出题或重新出答案，只负责审题、汇总、统计、看板、断点续跑、质量校准和重出内容整理。

## 什么时候使用

当用户需要审核一批 AI 面试题时，使用本 Skill。典型场景包括：

- 用户需要审核一个或多个岗位的大批量题库。
- 审核流程需要多个模型并行审题。
- 审核需要一轮、二轮、三轮到 N 轮逐步筛题。
- 用户需要统计每轮删除题目的错误类型、错误数量和典型例子。
- 用户希望看到每个模型、每个批次的进度，及时发现卡住、过严或过松的问题。

## 开始前需要确认的信息

开始审题前，需要向用户确认以下信息：

1. 待审题表文件路径。
2. 当前要执行的审核轮次，例如一轮、二轮、三轮或第 X 轮。
3. 本次审核使用的模型名单；如果用户未明确给出，先请用户确认后再开始。

默认按每 `40` 题一批执行，不再向用户额外询问批量大小。

不要向用户询问行业和岗位，行业、岗位、招聘场景、技能、知识点等信息应从题目表格中读取。

不要向用户询问审题提示词来源，提示词已经固定在本 Skill 的正文流程中。

## 一轮审核

### 目标

第一轮中被任一模型标注删除的题，不进入第二轮。它们的删除结论、错误类型和删除原因保留在一轮审核结果汇总表中，后续结合第二轮结果一起整理重出内容。

## 二轮审核

### 目标

第二轮只审核第一轮中所有模型都没有审出错误的题目。第二轮重复一轮的多模型审题流程，用于复查第一轮放行题目的质量。

第二轮结束后，结合前两轮审核结果，整理：

- 可以保留的题目和答案。
- 需要保留题干、重新出答案的题。
- 需要题目和答案都重新出的题。
- 直接由脚本输出 3 个独立表：`行业-岗位-可保留的题目-2轮`、`行业-岗位-需要重新出题-2轮`、`行业-岗位-重新出答案-2轮`。表①列名和当前保持一致；表②、表③分别按指定模板重构，模板中的 `提示词` 和 `JD` 列不填。

- `GPT5.4`

- `Opus4.7`

## 三到 N 轮审核

### 目标

三到 N 轮审核用于处理“重出后回流”的题目或答案。下游重新生成后，用户会把新的题目或答案交回给本 Skill，本 Skill 继续审核这些回流内容。

三到 N 轮按“审核 -> 剔除问题题 -> 整理重出内容 -> 用户回流重出结果 -> 再审核”的方式循环，直到仍有问题的题目少于 `10` 个，再提示用户进入人工判断。

### 模型

三到 N 轮使用 2 个模型分别审题：

- `GPT5.5`
- `Gemini3.1pro`

- `X轮审核结果`
- `X轮审核结果汇总`
- `X轮审核后需要重出的内容`

### 输入文件

## 通用执行规则

### 归档与目录规则

审核过程中必须先按 `行业/岗位` 建立根目录，再在对应岗位目录下创建本轮需要的所有文件夹、批次文件、模型结果、汇总文件和看板文件，避免不同岗位的内容混放。

- 先确定当前题库对应的 `行业` 和 `岗位`，再创建该岗位专属工作目录。
- 所有本轮生成的文件都必须放在对应的行业-岗位目录内，不得写到共享根目录下。
- 若同一行业下有多个岗位，必须继续按岗位拆分，禁止混用。
- 后续每轮回流、重出和复审文件，也必须回到同一行业-岗位目录链路中管理。

### 分批规则

按岗位分别整理待审题表，并保持原始题目顺序稳定不变。

- 默认每 `40` 题为一批。
- 批次编号使用固定格式：`R{轮次}-B{三位批次号}`，例如 `R1-B001`。
- 不要抽样审题，每一题都必须进入审题流程。

### 主索引字段

准备批次时，主索引至少保留以下信息：

- `question_id`
- `招聘场景` 或 `场景`
- `行业`
- `岗位` 或 `岗位名称`
- `batch_id`
- `题目` 或 `问题`
- `参考答案-第一层` 或 `第一层`
- `参考答案-第二层` 或 `第二层`
- `参考答案-第三层` 或 `第三层`
- `难度`
- `建议作答时长` 或 `建议作答时间`
- `技能分类`
- `技能`
- `知识点`

字段映射规则：

- `题目` 可映射为 `问题`。
- `参考答案-第一层` 可映射为 `第一层`。
- `参考答案-第二层` 可映射为 `第二层`。
- `参考答案-第三层` 可映射为 `第三层`。
- `岗位` 可映射为 `岗位名称`。
- `招聘场景` 可映射为 `场景`。
- `建议作答时长` 可映射为 `建议作答时间`。

### Pilot Gate（质量校准）

大批量任务不要一开始就跑完全量。每个模型先试跑 `1-2` 个批次，建议试跑 `2` 个批次，也就是最多 `80` 题。

试跑结束后检查：

- 每个模型的删题率是否明显过高或过低。
- 每个模型的删题率与其他模型中位数相比是否偏差过大。
- 模型给出的删除理由是否符合审题提示词和错误分类口径。
- 模型保留题目的理由是否合理，是否存在明显漏审。

如果某个模型明显过松、过严或误解规则，应暂停该模型后续批次，先校准提示词，再重新跑 pilot。不要在 pilot 未通过时直接进入全量审核。

### 模型审题结果字段

每个模型的审题结果必须在原表字段基础上追加以下字段：

- `是否删除`
- `错误类型`
- `删除原因`
- `疑似与肯定`
- `删除轮次`（第二轮及后续轮次必须填写）

字段填写要求：

- `是否删除` 只能填写 `是` 或 `否`。
- `错误类型` 优先填写二级错误类型，例如 `A1~A4`、`B1~B6`、`C1~C2`。如果无法判断二级类型，才退回到 `A/B/C`。
- `删除原因` 必须写清楚具体错在哪里，以及正确应该是什么；如果是重复题，按提示词要求可以不填删除原因。
- `疑似与肯定` 只能填写 `肯定有`、`疑似有`、`肯定无`、`疑似无`。
- `删除轮次` 填写当前审核轮次，例如 `2`、`3`；第一轮审核结果可以不填。

### 标准化与合并规则

每轮汇总前，必须先统一 `疑似与肯定` 和 `是否删除`：

- `疑似与肯定` 为 `肯定有` 或 `疑似有` 或 `疑似无` 时，统一视为 `是否删除 = 是`。
- `疑似与肯定` 为 `肯定无` 时，统一视为 `是否删除 = 否`。

同一题在多模型汇总时：

- 只要任一模型标记为 `是`，本轮合并结果就视为有问题。
- 只有所有模型都标记为 `否`，本轮合并结果才视为无问题。

### 重出分流规则

本 Skill 只整理重出内容，不执行重出。

- 仅出现 `B` 类错误的题目，进入 `保留题干重出答案`，表示题干可以保留，只需要重新出参考答案。
- 出现 `A` 类或 `C` 类错误的题目，进入 `需要重新出题`，表示题目和答案都需要重新出。
- 无错误题目进入 `可保留题目答案`。

### 题目数量守恒校验

每个审核任务必须建立题目数量守恒校验，避免回流重出结果被简单拼接后导致题目数膨胀。

- 第一轮开始时记录 `initial_total`，等于原始待审核题表行数。
- 从第二轮开始，每轮审核结束后的题库状态总数必须等于 `initial_total`：`可保留题目答案` + `需要重新出题` + `保留题干重出答案` = `initial_total`。
- 每轮进入下一轮的待审核题数必须等于上一轮待回流题数：`需要重新出题` + `保留题干重出答案` = `下一轮待审核题目总数`。
- 重新出题和保留题干重出答案是两类任务，不能把用户回流的多个 Excel 直接纵向拼接为下一轮输入；必须以上一轮 `题库审核汇总.xlsx` 中的 `需要重新出题` 和 `保留题干重出答案` 为主索引，一题一条地匹配回流结果。
- 如果回流文件合计行数、匹配后的任务数或下一轮待审核表行数与上一轮待回流题数不一致，必须暂停，不得进入模型审核，并向用户报告缺失、重复或多出的题目。
- 第三轮到第 N 轮如果只审核待回流题，仍需维护一个完整题库状态台账：已保留题继续计入 `可保留题目答案`，本轮仍有问题的题进入重出分流，三类状态合计必须始终等于 `initial_total`。

从第二轮开始，以及第三轮到第 N 轮每轮结束后，优先输出一个带审核轮次的题库审核汇总表，例如 `2轮题库审核汇总.xlsx`、`3轮题库审核汇总.xlsx`，包含三个 sheet：

1. `可保留题目答案`
2. `需要重新出题`
3. `保留题干重出答案`

三个 sheet 的字段参考样例文件，优先使用：

- `问题`
- `第一层`
- `第二层`
- `第三层`
- `技能分类`
- `技能`
- `知识点`
- `难度`
- `行业`
- `岗位名称`
- `场景`
- `一审_是否删除`
- `一审_删除原因`
- `二审_是否删除`
- `二审_删除原因`
- `题目是否修改`
- `是否重新出题`
- `一审_错误大类集合`
- `二审_错误大类集合`

第三轮到第 N 轮可以将 `一审/二审` 字段按实际轮次扩展为 `X轮_是否删除`、`X轮_删除原因`、`X轮_错误大类集合`，但不要改动用户明确提供的模板字段。

1. `行业-岗位-可保留的题目-N轮`
2. `行业-岗位-需要重新出题-N轮`
3. `行业-岗位-重新出答案-N轮`

其中：

## 每轮统计与看板

每轮都必须输出统计结果和进度看板。

统计内容包括：

- 删除题目的一级错误类型数量：`A/B/C`。
- 删除题目的二级错误类型数量：例如 `A1/A2/B1/B2/B3/C2`。
- 每个二级错误类型给出 `2` 个代表性例子。
- 每个例子应包含题目、错误表现和删除原因，不要截断题干或理由。

看板只保留两个文件：

- `round{轮次}_dashboard.html`
- `round{轮次}_dashboard.xlsx`

不要单独输出细分错误统计看板，例如不要保留 `*_删除错误统计_细分.html`、`*_删除错误统计_细分.xlsx` 或 `*_删除错误统计_细分.md`。细分错误统计应嵌入 dashboard 内。

## 进度日志要求

每个模型、每个批次都要记录日志，字段包括：

- `round`
- `model`
- `batch_id`
- `status`
- `start_time`
- `end_time`
- `duration_sec`
- `questions_total`
- `delete_count`
- `keep_count`
- `error_A_count`
- `error_B_count`
- `error_C_count`

`status` 只能使用：

- `pending`
- `running`
- `done`
- `blocked`

如果模型卡住，必须在日志中写入 `blocked`，并在 `notes` 中说明原因。

## 卡住处理规则

如果 10 分钟内没有任何批次状态更新，应视为疑似卡住。此时需要输出：

1. 当前各模型的 `done/running/pending/blocked` 批次数。
2. 每个模型最后完成的批次。
3. 每个模型建议从哪个批次继续。

恢复时必须保证幂等：已经完成的批次不重复写入、不重复汇总。

## 模型质量漂移处理规则

在 pilot gate 和长批次审核过程中，都要关注模型是否过严或过松。

触发告警的情况包括：

- 删除率低于 `5%`，可能过松。
- 删除率高于 `45%`，可能过严。
- 某模型删除率与同组模型中位数相差超过 `15` 个百分点。
- 删除理由反复不符合审题提示词或错误分类定义。

如果触发质量告警，需要：

1. 暂停该模型后续批次。
2. 展示该模型 5 个保留样例和 5 个删除样例。
3. 说明该模型可能误解了哪条规则。
4. 校准提示词后重新跑 pilot。
5. 只有重新通过 pilot 后，才继续全量审核。

## 与用户沟通时的汇报顺序

向用户汇报时，按以下顺序输出：

1. 当前轮次的总体进度。
2. 是否有卡住、失败或质量漂移风险。
3. 删除题目的一级错误类型统计。
4. 删除题目的二级错误类型统计。
5. 每个二级错误类型的 2 个典型例子。
6. 下一步动作，以及是否需要用户介入。

如果当前存在卡住或质量漂移，还要追加：

7. 每个模型应从哪个批次继续。
8. 每个模型是否需要重新 pilot。

## 可用脚本

辅助脚本位于 `scripts/` 目录。每轮汇总优先使用脚本完成，只有脚本字段不匹配、输入文件异常或用户要求临时处理时，才由 Agent 手工整理。

- `merge_round1_results.py`：合并一轮模型结果，导出一轮汇总、待二轮审核题表和错误统计。
- `merge_round2_results.py`：合并二轮模型结果，导出二轮汇总、题库审核汇总和错误统计。
- `merge_roundx_results.py`：合并三到 N 轮模型结果，导出 X 轮汇总、题库审核汇总和错误统计。
- `review_merge_utils.py`：汇总脚本公共工具，不直接运行。
- `export_dashboard_views.py`：根据日志和错误统计导出 `dashboard.html` 和 `dashboard.xlsx`。
- `build_detailed_error_stats.py`：从模型结果中生成细分错误统计；如果已使用新合并脚本的 `*_error_stats.xlsx`，可不再单独运行。
